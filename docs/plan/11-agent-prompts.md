# 구현 에이전트 지시 프롬프트 예시

> 목적: 각 단계를 다른 에이전트에게 넘길 때 **문맥 이탈로 인한 재작업 루프**를 막는다.
> 사용법: 아래 **공통 헤더**를 모든 에이전트에 똑같이 넣고, **단계별 본문**만 바꿔 붙인다.
> ⚠️ 계획 전체를 한 프롬프트로 전 에이전트에 뿌리지 말 것. 웨이브 순서(맨 아래)를 사람이 통제할 것.

---

## 0. 공통 헤더 (모든 에이전트에 동일하게 삽입)

```
[역할]
너는 teamd-backend(Kotlin + Spring Boot 단일 프로세스)의 한 단계를 구현하는 에이전트다.

[불변 규칙 — 어기면 멈추고 보고]
- 진실의 원본은 docs/specs/ 의 4개 문서다: 기능명세서_v1.3, API명세서_v1.1, ERD_v1.0, 시스템아키텍처_v1.0.
  구현과 문서가 어긋나면 구현으로 덮어쓰지 말고, 멈추고 어긋난 지점을 보고하라.
- backend/AGENTS.md 의 Kotlin/JPA/DTO/보안/관측성 규칙을 반드시 따르라.
- 네 작업 범위는 아래 "이번 작업" 한 단계뿐이다. 그 밖의 엔드포인트·도메인·파일을 건드리지 마라.
- 선행 단계 산출물이 없으면 직접 만들지 말고 "선행 미완료: <무엇이 없는지>"로 보고하고 멈춰라.
- 완료 기준 = 해당 단계 문서의 "✅ 검증 게이트" 전 항목 통과 + `./gradlew build` 그린.
  이 기준을 넘어선 리팩터링·추상화·추가 기능을 만들지 마라. (오버엔지니어링 금지)
- 외부 API 실제 키로 테스트하지 마라. 모든 테스트는 stub/Testcontainers로 한다.
- 커밋 메시지는 한국어, 하나의 논리 단위만.

[문맥 자료]
- 계획 목차: backend/docs/plan/00-INDEX.md
- 공통 검증 정책: backend/docs/plan/09-verification-policy.md
- FE 계약(응답 형태 주의): backend/docs/plan/10-fe-contract-sync.md

[산출 형식]
1) 변경 파일 목록  2) 각 검증 게이트 항목별 통과 근거(테스트 이름)  3) 문서와 어긋나 보고할 사항(없으면 "없음")
```

## 1. 단계별 본문 (공통 헤더 아래에 이어 붙임)

### P0 — 기반 (⚠️ 단독·최우선, 병렬 금지)

```
[이번 작업] P0 — 기반 공사
- 정독: backend/docs/plan/01-phase0-foundation.md (네 명세)
- 선행: 없음. 현재 코드는 H2 + 인증 없음 상태다.
- 핵심: H2→PostgreSQL 전환, Flyway V1 baseline(ERD 전체), 인증 필터, 좌표 암호화,
  공통 오류/페이지네이션 규약, RestClient 사용(WebFlux 금지).
- 이 산출물은 이후 모든 단계가 공유한다. 인터페이스를 임의로 바꾸지 마라.
- 완료 후 V0-1~V0-10 근거 제출.
```

### P1 — 보드·초대·참여자

```
[이번 작업] P1 — 보드·초대·참여자 (엔드포인트 1~8)
- 정독: backend/docs/plan/02-phase1-board-participant.md
- 선행 전제: P0가 머지되어 있다(BaseEntity/인증 필터/PublicId/오류규약 존재). 없으면 보고하고 멈춰라.
- 주의: 초대 코드·공개 토큰은 원문 저장, 참여 토큰 secret만 HMAC.
  타인 출발지는 registered 불리언만 노출. CLOSED 보드 쓰기는 409.
  엔드포인트 8의 출발지 변경 잠금(T1-4b)은 FOR UPDATE 행 잠금으로 구현.
- 완료 후 V1-1~V1-9 근거 제출.
```

### P2 — 장소 검색·등록

```
[이번 작업] P2 — 장소 검색·등록 (엔드포인트 9~15)
- 정독: backend/docs/plan/03-phase2-place-search.md
- 선행 전제: P0·P1 머지. 없으면 보고하고 멈춰라.
- 주의: 외부 호출 공통 기반(타임아웃/429 백오프/오류매핑/로그 마스킹)을 여기서 레퍼런스로 만든다.
  검색만으로 Place를 만들지 말고 TMAP/ODsay를 호출하지 마라.
  PlaceUsageChecker 인터페이스를 정의만 하고 구현체는 두지 마라(P3/P4가 각자 추가).
- 완료 후 V2-1~V2-11 근거 제출.
```

### P3 — 댓글·투표 (P4·I2와 병렬 가능)

```
[이번 작업] P3 — 댓글·투표 (엔드포인트 16~24)
- 정독: backend/docs/plan/04-phase3-comment-vote.md
- 선행 전제: P2 머지. 없으면 보고하고 멈춰라.
- 파일 충돌 주의: VotePlaceUsageChecker는 반드시 새 파일로만 추가하고 P2/P4 파일을 수정하지 마라.
  열린 투표 1개 제약은 부분 unique 인덱스 + 예외 변환으로. 익명 투표는 신원 미노출.
  투표 PUT 교체는 단일 트랜잭션 delete→insert.
- 완료 후 V3-1~V3-12 근거 제출.
```

### P4 — 코스·공개 일정 (P3·I2와 병렬 가능)

```
[이번 작업] P4 — 코스 초안·확정·공개 일정 (엔드포인트 27~31, 34)
- 정독: backend/docs/plan/05-phase4-course-public.md
- 선행 전제: P2 머지. 없으면 보고하고 멈춰라.
- 파일 충돌 주의: CoursePlaceUsageChecker는 새 파일로만 추가.
  코스 초안은 If-Match/ETag 낙관적 잠금(헤더 없으면 400, 불일치 412).
  공개 일정 응답에서 참여자/출발지/토큰/댓글/투표/boardId 제외. CLOSED면 404.
- 완료 후 V4-1~V4-14 근거 제출.
```

### P5 — 출발 안내

```
[이번 작업] P5 — 개인 출발 안내 (엔드포인트 32~33)
- 정독: backend/docs/plan/06-phase5-departure.md
- 선행 전제: P4(확정 코스)·P1(출발지) 머지. 없으면 보고하고 멈춰라.
- 주의: 여기서 Job Executor 뼈대를 만든다(P6가 재사용). 비동기 스택은 @Scheduled + DB 폴링 +

  경로 없음은 UNAVAILABLE 상태로만 수렴(422 아님). 외부 호출 중 DB 트랜잭션 열지 마라.
- 완료 후 V5-1~V5-14 근거 제출.
```

### P6 — 지역 찾기 (마지막)

```
[이번 작업] P6 — 만나기 좋은 지역 찾기 (엔드포인트 25~26)
- 정독: backend/docs/plan/07-phase6-area-search.md
- 선행 전제: P5(Job Executor)·P1(출발지) 머지 + I1의 ODsay 고정 IP 등록(운영 검증용). 없으면 보고.
- 주의: snapshot에 참여자 ID만 저장(좌표 복사 금지). 좌표 일관성은 FOR UPDATE 행 잠금으로.
  POST는 대상 참여자 행을 ID 오름차순 FOR UPDATE→검증→job 생성. 동시 POST/PATCH 테스트 필수.
  실패코드에 ORIGIN_REQUIRED 포함(POST=422, 접수 후 소실=job FAILED.error.code). JTS 사용, Node.js 금지.
- 완료 후 V6-1~V6-18 근거 제출.
```

## 2. 디스패치 순서 (사람이 통제)

**동시에 전부 뿌리지 말 것.** 각 웨이브는 앞 웨이브의 검증 게이트 통과 + 머지 확인 후 시작한다.

| 웨이브 | 동시 착수 가능 | 게이트 |
|---|---|---|
| 0 | **P0 단독** (병렬 금지) | V0 전항목 + 머지 |
| 1 | P1 / I0 | 각 검증 게이트 + 머지 |
| 2 | P2 / I1 | V2 / VI1 + 머지 |
| 3 | P3 / P4 / I2 | 각 게이트 + 머지 |
| 4 | P5 / I3 | V5 / VI3 + 머지 |
| 5 | P6 | V6 전항목 |

**파일 충돌 방지 규칙 (같은 웨이브 내부):**
- P3·P4 병렬 → `PlaceUsageChecker` 구현체는 각자 **새 파일**로만. 기존 파일 수정 금지
- 엔드포인트 8 출발지 잠금은 P1만, P6는 POST 쪽 잠금만 (같은 계약을 양쪽에서 각자 구현)

## 3. 루프가 시작되면 (사람 개입 신호)

- 에이전트가 "선행 미완료"를 보고하면 → 순서가 틀린 것. 선행 단계부터 머지
- 검증 게이트를 못 넘고 3회 이상 같은 곳을 고치면 → 명세 해석이 어긋난 것. `docs/specs/` 원문을 직접 인용해 재지시
- 두 에이전트가 같은 파일을 고치면 → 웨이브/파일 분리 규칙 위반. 한쪽을 멈추고 인터페이스 분리 재확인
```
