# 공통 검증 정책

> 각 단계 문서의 "✅ 검증 게이트"는 이 문서의 규칙을 전제로 한다.

---

## 1. 테스트 레이어

| 레이어 | 대상 | 도구 | 언제 |
|---|---|---|---|
| 단위 | 계산식·매핑·암복호·토큰 해시·JTS | JUnit5 + kotlin-test | 로직이 있는 모든 곳 |
| 영속 | 엔티티 매핑, JSONB 왕복, 부분 unique 인덱스 | `@DataJpaTest` + **Testcontainers PostgreSQL** | 스키마를 건드릴 때 |
| 계약 | HTTP 상태코드·오류코드·응답 필드 | `@SpringBootTest` + MockMvc + Testcontainers | **모든 엔드포인트 필수** |
| 외부 어댑터 | 429 백오프, 오류 매핑, 파싱 | stub HTTP 서버 | 외부 API를 붙일 때 |
| 운영 스모크 | 배포 직후 최소 흐름 | 스크립트 | 배포마다 |

**H2로 테스트하지 않는다.** JSONB·부분 unique 인덱스·`SKIP LOCKED`는 PostgreSQL에서만 정확히 동작한다.

---

## 2. 계약 테스트 최소 요건

각 엔드포인트마다 **최소 3개**:
1. 정상 경로 (상태코드 + 핵심 응답 필드)
2. 권한 실패 (401 또는 403 또는 404)
3. 그 엔드포인트 고유의 도메인 실패 (409 / 412 / 422 …)

응답 검증은 "200이 왔다"가 아니라 **명세의 필드명·오류코드 문자열까지** 비교한다.

## 3. 보안 체크리스트 (모든 단계 공통, 매 PR)

| # | 항목 |
|---:|---|
| S-1 | 응답에 참여 토큰이 포함되는 곳은 보드 생성·참여 **2곳뿐** |
| S-2 | 타인의 출발지 좌표·라벨이 어떤 응답에도 없음 |
| S-3 | 로그에 토큰·초대 코드·공개 토큰·API 키·좌표·검색어 원문 없음 |
| S-4 | 오류 `details`에 외부 API 본문·스택트레이스·SQL 없음 |
| S-5 | 공개 API 응답에 참여자·댓글·투표·내부 boardId 없음 |
| S-6 | CORS에 `*` 없음 (운영 프로필) |
| S-7 | 새로 추가한 외부 URL 저장 경로에 도메인 검증 있음 |
| S-8 | 출발지 좌표의 평문 사본이 새로 생기지 않음 (`participant.origin_ciphertext` 외 저장 경로 없음) |
| S-9 | 비밀값이 git·컨테이너 이미지·로그에 없음 (`.env` 미커밋, 이미지 레이어 스캔) |

## 4. 단계 완료 판정

```
[ ] ./gradlew build 그린
[ ] 해당 단계 계약 테스트 전부 통과
[ ] API명세서 16절의 해당 E2E 시나리오 통과
[ ] 보안 체크리스트 S-1~S-9 통과
[ ] 명세와 어긋난 부분을 발견했다면 → 구현 변경이 아니라 이슈로 보고했는가
```

하나라도 미충족이면 **다음 웨이브를 시작하지 않는다.**

## 5. MVP 최종 검증 (전 단계 완료 후)

API명세서 17절 10개 항목을 그대로 체크리스트로 사용한다.

| # | 기준 | 검증 방법 |
|---:|---|---|
| 1 | 모든 인증 API가 보드 범위·역할 검증 | 교차 보드 접근 테스트 |
| 2 | 사용자가 후보를 선택하기 전 Place 미생성 | 검색 호출 후 테이블 count |
| 3 | 장소 검색만으로 TMAP·ODsay 미호출 | stub 카운터 |
| 4 | 지역 찾기는 호스트 POST로만 실행 | 다른 경로에서 job 미생성 확인 |
| 5 | 출발 안내는 참여자 명시 요청으로만 | 코스 확정 후 자동 호출 없음 |
| 6 | 두 작업 모두 중복 생성 없음 | 동시 요청 테스트 |
| 7 | 첫 코스 장소만 FIRST_MEETING이며 출발 안내 목적지 | 초안 검증 + 목적지 일치 |
| 8 | 공개 API에 민감정보 미노출 | 응답 키 부재 assert |
| 9 | 외부 429·장애·예산 초과가 정의된 오류로 수렴 | stub 시나리오 3종 |
| 10 | NAVER 지역검색·외부 URL 가져오기·SSE·푸시·운영 API가 코드에 없음 | 소스 전역 grep |

## 6. 회귀 방지

- 버그를 고칠 때는 **재현 테스트를 먼저** 추가한다 (AGENTS.md)
- 명세 변경이 필요하면 `docs` repo의 스펙을 먼저 갱신하고, 그 다음 구현과 테스트를 바꾼다
- 각 단계의 계약 테스트는 이후 단계에서도 계속 실행된다. 깨지면 **그 단계로 돌아간다**
